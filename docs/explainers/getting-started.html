Before You Begin
Gather the four things the module needs before a single resource is created.
Required accounts & tools
☁️
Hetzner Cloud account + API token
Create a project in the Hetzner Cloud Console and generate a Read & Write API token. This is passed as hcloud_token.
🔑
SSH key in Hetzner project
Upload your public key to the Hetzner project (Security → SSH Keys). The key's name or ID is passed as ssh_keys. The private key contents go into ssh_private_key — the module SSHes to the first control plane node to retrieve the kubeconfig.
🛠️
Terraform >= 1.5 installed locally
The module uses optional() object attributes and required_version = ">= 1.5" is enforced in the module itself.
then set your variables
Sensitive variable handling
✓
Use terraform.tfvars Copy examples/minimal/terraform.tfvars.example to terraform.tfvars and fill in your token, SSH key name, and private key.
⚠
Never commit .tfvars Both hcloud_token and ssh_private_key are marked sensitive = true — they will not appear in plan output, but they are stored in Terraform state. Use an encrypted remote backend.
ℹ
Optional add-on tokens hcloud_ccm_token, hcloud_csi_token, and hcloud_autoscaler_token all default to hcloud_token if left null — safe for a first deploy.

02
2Apply
Two-Phase
Apply
Write the module call, run terraform init, then apply in two targeted phases to separate infrastructure from add-ons.
Minimal module call
📝
Write main.tf
Source the module, set cluster_name, hcloud_token, ssh_keys, and ssh_private_key. Add one entry to node_pools with a name, server_type, and node_count. See examples/minimal/main.tf for the complete starter file.
📦
terraform init
Downloads the hetznercloud/hcloud provider (>= 1.58.0) plus the hashicorp/null, hashicorp/random, hashicorp/tls providers used internally.
phase 1 — infrastructure
Phase 1 — provision & wait
🌐
terraform apply -target=module.cluster.null_resource.wait_for_cluster
Creates network, subnet, firewall, placement group, load balancer, 3 CP nodes, and worker nodes via cloud-init. Then SSH-polls the first CP until rke2-server is active and the API server responds (up to 15 min timeout).
📋
Kubeconfig fetched automatically
After wait_for_cluster, null_resource.fetch_kubeconfig SSHes to the first CP, cats /etc/rancher/rke2/rke2.yaml, rewrites the server address to the load balancer IP, and writes .kube/<cluster_name>.yaml locally with chmod 600.
phase 2 — add-ons
Phase 2 — deploy add-ons
🚀
terraform apply (no target)
Deploys Helm releases and Kubernetes resources via the module.addons submodule. The Helm and Kubernetes providers read the kubeconfig written in Phase 1. CCM and CSI are enabled by default; all other add-ons require explicit enable_* flags.
why two phases?
Why this split is necessary
ℹ
Provider config resolves at plan time The Helm and Kubernetes providers need a valid kubeconfig endpoint to initialize. On a first apply the cluster does not exist yet, so a single-phase apply would fail at provider configuration.
✓
Static first-CP IP breaks the circular dep Worker cloud-inits reference the first CP's private IP (cidrhost(cluster_subnet_cidr, 10)) at plan time, so workers can be provisioned in parallel without waiting for the CP's IP to be dynamically assigned.

3Verify
Post-Deploy
Checks
Export the kubeconfig, confirm nodes are Ready, and validate that the default add-ons are healthy.
Export kubeconfig
📤
terraform output -raw kubeconfig > ~/.kube/my-cluster.yaml
The kubeconfig output is marked sensitive; use -raw to write it without quotes. The server address is already rewritten to the load balancer IP (control_plane_lb_ip output).
🔗
export KUBECONFIG=~/.kube/my-cluster.yaml
Or merge into your default kubeconfig with KUBECONFIG=~/.kube/config:~/.kube/my-cluster.yaml kubectl config view --flatten.
confirm cluster health
Node readiness
✓
kubectl get nodes Expect 3 control plane nodes and all configured worker nodes in Ready status. All nodes run Ubuntu 24.04 and RKE2 v1.32.x by default.
✓
kubectl get nodes -l node-role=control-plane The module applies the label node-role=control-plane to all three CP nodes automatically.

what's next
Optional next steps
ℹ
Lock down API access Set kube_api_allowed_cidrs to your VPN or Tailscale CGNAT range instead of the default open 0.0.0.0/0.
ℹ
Enable add-ons incrementally Set enable_ingress, enable_cert_manager, enable_longhorn, or enable_flux to true and re-run terraform apply — no re-provisioning needed.
ℹ
See the full example examples/full/main.tf shows Helm and Kubernetes provider wiring for all add-ons enabled simultaneously.
